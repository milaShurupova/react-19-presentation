React Compiler

The Problem

  • In React 18, you manually optimize re-renders with useMemo, useCallback, and React.memo
  • Adds cognitive overhead — you have to think about dependencies constantly
  • Creates boilerplate — wrapping and unwrapping everywhere
  • Easy to get wrong — stale closures, missing deps, unnecessary re-renders
"Performance optimization shouldn't be the developer's job."
Compiler

React Compiler

The Solution

  • React Compiler (formerly "React Forget") — an optimizing build-time compiler
  • Automatically memoizes components, props, and hook values
  • Transforms your React code into more efficient JavaScript
  • Zero manual intervention needed

function ExpensiveList({ items, filter }) {
  const filtered = items.filter(i => i.name.includes(filter));

  const handleClick = (id) => {
    console.log('clicked', id);
  };

  return filtered.map(item => (
    <Item key={item.id} onClick={handleClick} {...item} />
  ));
}

Installing the React Compiler in Next.js

1. Install the compiler package

npm install babel-plugin-react-compiler

2. Enable it in next.config.js

// next.config.js
const nextConfig = {
  experimental: { reactCompiler: true },
};
export default nextConfig;
That's it — Next.js picks up the compiler automatically, no extra Babel config needed.

3. ESLint plugin

npm install eslint-plugin-react-compiler
// .eslintrc.json
{
  "plugins": ["react-compiler"],
  "rules": {
    "react-compiler/react-compiler": "error"
  }
}
✨ Memo badge in React DevTools

Optimized components show a ✨ Memo badge. ESLint error = compiler skipped it — code still works, just unoptimized. No crash.

Ref Cleanup Functions

React 18 — useEffect workaround
function MyComponent() {
  const [visible, setVisible] = useState(false);
  const ref = useRef(null);

  useEffect(() => {
    const observer = new IntersectionObserver(
      ([entry]) => setVisible(entry.isIntersecting)
    );
    observer.observe(ref.current);
    return () => observer.disconnect();
  }, []);

  return <div ref={ref} />;
}
React 19 — ref callback with cleanup
function MyComponent() {
  const [visible, setVisible] = useState(false);

  function trackVisibility(node) {
    const observer = new IntersectionObserver(
      ([entry]) => setVisible(entry.isIntersecting)
    );
    observer.observe(node);
    return () => observer.disconnect();
  }

  return <div ref={trackVisibility} />;
}
✓ No separate useEffect
✓ Cleanup runs on unmount
✓ Reusable callback

import { prefetchDNS, preconnect,
         preload, preinit } from 'react-dom';

function MyComponent() {
  preload('/font.woff', { as: 'font' });
  preconnect('https://api.example.com');
  prefetchDNS('https://cdn.example.com');
  preinit('/script.js', { as: 'script' });
}

<Activity>

Hide and restore UI without unmounting — preserve state across visibility changes

<Activity mode="visible" | "hidden">{children}</Activity>

The Problem

  • {show && <C />} → component unmounts when hidden
  • State destroyed: form inputs, scroll, expanded sections
  • Re-mount = re-fetch data + re-run effects (slow)

The Solution

  • mode="hidden" → just CSS display: none
  • Component stays mounted — all state alive
  • Effects pause when hidden, resume on reveal
  • Can pre-render hidden content in background
When to Use — Tab UIs, sidebars/drawers, navigation stacks, modals with form state that must survive toggling
When NOT to Use — Simple conditional renders with no meaningful state. Large hidden subtrees stay in memory + their DOM remains in the page.

Actions — Old vs New

The same form, two eras of React

React 18 — manual

  • onSubmit handler on <form>
  • e.preventDefault() — stop browser reload
  • fetch() — manually send data
  • useState × 3 — isPending, error, result
  • try/catch — handle errors
20+ lines of boilerplate. Per form.

React 19 — automatic

  • async fn to <form action={...}>
  • React handles transition automatically
  • No e.preventDefault() — React intercepts
  • No manual useState for loading / error
  • FormData — uncontrolled inputs
Half the code. Zero boilerplate.
Key insight — Actions are wrapped in useTransition under the hood. The rest of the UI stays interactive during submission.

Forms — React 18 vs React 19

Same scenario: a form with one input that calls updateName()

React 18 — manual wiring
function UpdateName() {
  // 3 manual state slots
  const [name, setName] = useState('');
  const [error, setError] = useState(null);
  const [isPending, setIsPending] = useState(false);

  const handleSubmit = async (e) => {
    e.preventDefault();         // stop reload
    setIsPending(true);         // flip on
    const error = await updateName(name);
    setIsPending(false);        // flip off
    if (error) setError(error);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input value={name}
        onChange={(e) => setName(e.target.value)} />
      <button disabled={isPending}>Update</button>
      {error && <p>{error}</p>}
    </form>
  );
}
React 19 — built-in
function UpdateName() {
  // Action — async fn, receives FormData
  const updateAction = async (formData) => {
    const error = await updateName(
      formData.get("name")
    );
    if (error) {
      // see useActionState →
    }
  };

  return (
    <form action={updateAction}>
      <input type="text" name="name" />
      <button type="submit">Update</button>
    </form>
  );
}

// What's gone vs React 18:
//   • 3× useState
//   • e.preventDefault()
//   • manual loading toggle
//   • controlled inputs

Question “How do I save the result of the last action call — and track its pending state — without piling up useState?”

useActionState

Manages the full lifecycle of a form action: previous state, current result, pending state

const [state, formAction, isPending] = useActionState(fn, initialState)

What it replaces

  • Manual useState for form results
  • useReducer patterns for form state
  • Custom pending state tracking

Returns 3 values

  • state — current action result
  • formAction — pass to <form action>
  • isPending — submission in progress
Login forms — server validation errors + pending state in one hook
Multi-step wizards — previous state tracks completed steps
CRUD forms — replaces useReducer + useState + useTransition
Data

useActionState in Practice

function ChangeName() {
  // 1. Define the action — gets (previousState, formData)
  const updateAction = async (previousState, formData) => {
    const error = await updateName(formData.get("name"));
    if (error) return error;   // ← becomes the new state
    redirect("/path");
    return null;                // success — clears state
  };

  // 2. Hand it to the hook — get [last result, wrapped action, pending]
  const [error, submitAction, isPending] = useActionState(updateAction, null);

  return (
    <form action={submitAction}>     // wrapped action, not raw fn
      <input type="text" name="name" />
      <button disabled={isPending}>Update</button>  // auto pending
      {error && <p>{error}</p>}     // auto state from hook
    </form>
  );
}
No useState, no try/catch, no manual pending toggle — just return values from the action.
Data
Question “How does a child component inside a form know that the form is currently submitting — without prop drilling?”

useFormStatus

Read the parent form's submission state from any child component — no prop drilling

function SubmitButton() {
  const { pending } = useFormStatus();   // reads nearest parent <form>
  return (
    <button type="submit" disabled={pending}>
      {pending ? "Saving..." : "Save"}
    </button>
  );
}

// Usage — drop it inside any <form>, no props needed:
<form action={updateAction}>
  <input name="name" />
  <SubmitButton />
</form>
Why — Without it, you must pass isPending as a prop from every parent form to every submit button
When — Design system buttons, shared form components, any reusable UI inside a <form>

Question “How do I show the new state to the user immediately — before the server actually replies?”

useOptimistic

Update the UI instantly while waiting for the server response

const [optimistic, addOptimistic] = useOptimistic(state, updateFn)

How it works

  • Shows the expected result immediately
  • Reverts automatically if the action fails
  • Merges with real data when server responds

Great for

  • "Like" buttons
  • Comment submission
  • Todo lists
  • Any action where instant feedback matters
Data

useOptimistic in Practice

// 1. Define the merge function — (currentState, newValue) => nextState
const addPending = (state, newMessage) => [
  ...state,
  { text: newMessage, sending: true }   // flag: show as "in flight"
];

// 2. Hand it to the hook with the real state
const [optimisticMessages, addOptimisticMessage] = useOptimistic(messages, addPending);

// 3. In the form action:
async function sendMessage(formData) {
  const message = formData.get("message");
  addOptimisticMessage(message);   // UI updates instantly
  await deliverMessage(message);  // real server call — auto-merge or revert
}
Demo tip: Add a setTimeout(resolve, 2000) to exaggerate the delay — shows the instant update vs real data arriving.

Action + 3 Hooks — Together

Where each hook lives · how they compose in real code

<form action={action}>
   ▲
   └─ useActionState
      spine: state + action + isPending

  <ul>{optimistic.map(...)}</ul>
   ▲
   └─ useOptimistic
      instant UI before server

  <input />
  <SubmitButton />
                  ▲
                  └─ useFormStatus
                     pending in child
</form>
function MessageForm({ messages }) {
  const [optimistic, addOptimistic] = useOptimistic(messages, addPending);

  // The Action — plain async fn (the foundation)
  const sendAction = async (prev, formData) => {
    const text = formData.get("message");
    addOptimistic(text);                  // show instantly
    return await deliverMessage(text);   // real server call
  };

  const [error, action, isPending] = useActionState(sendAction, null);

  return (
    <form action={action}>
      <ul>{optimistic.map(m => <li>{m.text}</li>)}</ul>
      <input name="message" />
      <SubmitButton />
    </form>
  );
}
  • Built on Action — <form action={asyncFn}> is the foundation; the 3 hooks just layer on top.
  • Together they replace the entire useState + useEffect + onSubmit pattern for forms.

Server Components

Components that run exclusively on the server — stabilized in React 19

Zero Client JS

Never shipped to the client bundle

Direct Data Access

Databases, files, server-only APIs

Better Performance

Smaller bundles, streamed HTML

SEO Friendly

HTML before client JS loads

"use client" — the boundary

  • First line of file — marks Client Component
  • Without it = Server Component (default)
  • Server can import Client — not the reverse (bridge via props/children)

Server Components can't use

  • useState, useEffect, useRef
  • Event handlers (onClick, etc.)
  • Rendered once on server — not "alive" in browser

Client Component → when to use

  • Forms & inputs
  • Like / favorite buttons
  • Modals, dropdowns, tooltips
  • Anything with animations or local state

Server Component → when to use

  • Product pages, blog posts
  • Dashboards (data-heavy)
  • Catalog & listing pages
  • Layout shells & navigation

Server Components — In Practice

No useState, no useEffect, no client bundle cost — just async rendering

// Never ships JS to the browser — runs once on server
async function ProductPage({ id }) {       // ← async component (new!)
  const product = await db.products.findUnique({ where: { id } });
  const reviews = await db.reviews.findMany({ where: { productId: id } });
  // ↑ direct DB access — no fetch, no API route, no useEffect

  return (
    <main>
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <AddToCartButton id={id} />  {/* Client Component — "use client" inside */}
      <ReviewList reviews={reviews} />
    </main>
  );
}
Use for — data-heavy pages, product listings, dashboards, blog content
Skip for — interactive widgets, forms, anything needing useState / useEffect

Server Actions

Functions you write locally that execute on the server — no manual API layer. Think: RPC for React.

  • "use server" directive marks an async function as server-only
  • Import & call from a Client Component — framework wires the network call behind the scenes
  • Plugs into <form action={serverFn}> for mutations
  • Replaces custom REST / GraphQL endpoints for common CRUD
Behind the scenes — the magic
"use client"
import { createNote } from "./actions";

function MyForm() {
  return <button onClick={() =>
    createNote({ text: "Hello" })
  }>Save</button>;
}
  1. Framework rewrites createNote(...) into a POST to a hidden URL like /_action/abc123
  2. Server receives the request & runs the real function
  3. Result is serialized and sent back
  4. You get a normal Promise — as if calling a local function
Framework support — Next.js App Router, Remix / React Router 7, TanStack Start

Putting It Together

Server Component + Server Action + Client Component on a single page

page.jsx — SERVER COMPONENT (no directive)
async function NotesPage() {              // async OK
  const notes = await db.notes.findMany();    // direct DB

  async function createNote(formData) {  // ← Server Action
    "use server";
    await db.notes.create({
      text: formData.get("text")
    });
  }

  return (
    <main>
      <ul>{notes.map(n => <li>{n.text}</li>)}</ul>
      <NoteForm action={createNote} />  // Client Comp
    </main>
  );
}
NoteForm.jsx — CLIENT COMPONENT
"use client"                          // boundary
import { useFormStatus } from "react";

function NoteForm({ action }) {        // receives Server Action via prop
  const { pending } = useFormStatus();   // hook OK here
  return (
    <form action={action}>           // fires on submit
      <input name="text" />
      <button disabled={pending}>
        {pending ? "Saving..." : "Add"}
      </button>
    </form>
  );
}

Server Actions — Important Points

⚠ Security — they're public HTTP endpoints

The action looks like a local call, but the framework exposes it at a hidden URL. Anyone can hit it with curl. Treat it like any API route.

Always:   ✓ authenticate the caller   ✓ check ownership/permissions   ✓ validate input shape

Not a replacement for every API

Best for your own React app's mutations & simple CRUD. Public APIs for mobile apps, third-party integrations, webhooks — still need a real REST/GraphQL endpoint (stable contract, non-React clients).

Framework required

Server Components & Actions need RSC-aware infrastructure: Next.js App Router, Remix / React Router 7, TanStack Start, Waku. Plain Vite or CRA — not supported.

use(promise) — Reading Promises

The bridge: Server Component starts the promise, Client Component unwraps it — streaming SSR for free

// Server Component — starts the promise (no await!)
async function Page() {
  const messagePromise = fetchMessage();
  return (
    <Suspense fallback={<p>Loading...</p>}>
      <Message messagePromise={messagePromise} />
    </Suspense>
  );
}

// Client Component — unwraps it during render
"use client"
import { use } from "react";

function Message({ messagePromise }) {
  const content = use(messagePromise);   // suspends until resolved
  return <p>{content}</p>;
}
Replaces — useEffect + useState + loading/error booleans
Reach for it when — promise comes as prop, conditional async reads
Avoid — don't create promises inside render (new promise on every re-render → loop)
Data

use(Context) — Conditional Context

The key differentiator: use is NOT bound by the Rules of Hooks — can be called inside conditionals, loops, or after early returns

useContext — classic hook

  • Must be at the top level
  • Cannot be inside if / loops
  • Cannot be after early returns

use — new in React 19

  • Can be called in conditionals
  • Can be called in loops
  • Can be called after early returns
useContext — always reads
function Header({ user }) {
  const theme = useContext(ThemeContext);
  // ↑ runs every render, even when
  //   we end up not using it

  if (!user) return <Login />;

  return <h1 style={{ color: theme.color }}>
    {user.name}
  </h1>;
}
use — read only when needed
function Header({ user }) {
  if (!user) return <Login />;
  // ↑ early return is fine

  const theme = use(ThemeContext);
  // ↑ reads only if user exists

  return <h1 style={{ color: theme.color }}>
    {user.name}
  </h1>;
}
Why classic hooks have the rule — React tracks hooks by call order. A skipped useState would shift all later hooks → state corruption. use is special at the runtime level — safe to call conditionally.

Improved Error Reporting

  • Errors no longer logged twice — StrictMode double-invocation noise removed
  • Better stack traces in development — component tree shown, not just react-dom.js

Three hooks on createRoot

onCaughtError
Error Boundary caught it — fallback shown. Log to Sentry.
onUncaughtError
Nothing caught it — app crashed. Critical alert.
onRecoverableError
React auto-recovered (e.g. hydration mismatch). Track health.

Wire them at app root

import { createRoot } from "react-dom/client";

const root = createRoot(container, {
  onCaughtError: (err, info) => {
    sentry.capture(err);
  },
  onUncaughtError: (err) => {
    alertOncall(err);
  },
  onRecoverableError: (err) => {
    metrics.log("recovered");
  },
});

root.render(<App />);
