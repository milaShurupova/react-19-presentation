React Compiler

The Problem

  • In React 18, you manually optimize re-renders with useMemo, useCallback, and React.memo
  • Adds cognitive overhead — you have to think about dependencies constantly
  • Creates boilerplate — wrapping and unwrapping everywhere
  • Easy to get wrong — stale closures, missing deps, unnecessary re-renders
"Performance optimization shouldn't be the developer's job."
Compiler

React 18: Manual Memoization

Every optimization is your responsibility

const ExpensiveList = React.memo(({ items, filter }) => {
  const filtered = useMemo(
    () => items.filter(i => i.name.includes(filter)),
    [items, filter]
  );

  const handleClick = useCallback(
    (id) => { console.log('clicked', id); },
    []
  );

  return filtered.map(item => (
    <Item key={item.id} onClick={handleClick} {...item} />
  ));
});

React Compiler

The Solution

  • React Compiler (formerly "React Forget") — an optimizing build-time compiler
  • Automatically memoizes components, props, and hook values
  • Transforms your React code into more efficient JavaScript
  • Zero manual intervention needed

React Compiler — What Changes in Your Code?

✓ Safe to Remove

  • React.memo()
    Compiler optimizes re-renders automatically
  • useMemo() for derived values
    Compiler tracks dependencies for you
  • useCallback()
    Compiler stabilizes function references

⚠ Keep When

  • useMemo() guards expensive computation
    Encodes business logic, not just rendering perf
  • Code the compiler can't analyze
    Dynamic patterns, eval, non-standard meta-programming
Enable in all new projects. Adopt incrementally in existing ones — start with non-critical pages.

Ref Cleanup Functions

  • Ref callbacks can now return a cleanup function — like useEffect
  • Perfect for cleaning up observers, event listeners, or third-party integrations
  • TypeScript tip: use block statements to avoid implicit return issues
<div ref={(node) => {
  // setup
  const observer = new IntersectionObserver(callback);
  observer.observe(node);

  // cleanup (new in React 19)
  return () => observer.disconnect();
}} />

Why It Matters

In React 18, ref callbacks had no cleanup. You needed a separate useEffect to disconnect observers — and if the element remounted, the old one leaked.

When to Use

  • IntersectionObserver / ResizeObserver on elements
  • Third-party DOM libraries (charts, maps, editors)
  • Manual event listeners on specific nodes

Resource Loading APIs

New react-dom APIs for preloading resources — prevents layout shifts, improves load time

import { prefetchDNS, preconnect, preload, preinit } from 'react-dom';

function MyComponent() {
  preinit('https://.../script.js', { as: 'script' });
  preload('https://.../font.woff', { as: 'font' });
  preload('https://.../style.css', { as: 'style' });
  prefetchDNS('https://cdn.example.com');
  preconnect('https://api.example.com');
}
Fonts — preload custom fonts to eliminate FOUT and layout shifts
Scripts — preinit third-party scripts (analytics, widgets) early
Navigation — preload next-route assets on hover for instant transitions
APIs — preconnect to API domains to cut round-trip latency

<Activity>

Hide and restore UI without unmounting — preserve state across visibility changes

<Activity mode="visible" | "hidden">{children}</Activity>

The Problem

  • Conditional rendering ({show && <C />}) unmounts the component
  • State is destroyed — form inputs, scroll position, expanded sections
  • Re-mounting is expensive — data re-fetched, effects re-run

The Solution

  • mode="hidden" hides with CSS (display: none)
  • React & DOM state fully preserved
  • Effects cleaned up, re-mounted on reveal
  • Enables pre-rendering hidden content
When to Use — Tab UIs, sidebars/drawers, navigation stacks, modals with form state that must survive toggling
When NOT to Use — Simple conditional renders with no meaningful state. Large hidden subtrees stay in memory + their DOM remains in the page.
Data

<Activity> — Before & After

Conditional (state lost)
function App() {
  const [show, setShow] = useState(true);

  return (
    <>
      {/* Unmounts → state lost */}
      {show && <Sidebar />}

      <button onClick={() =>
        setShow(!show)
      }>
        Toggle
      </button>
    </>
  );
}
Activity (state preserved)
function App() {
  const [show, setShow] = useState(true);

  return (
    <>
      {/* Hidden via CSS → state kept */}
      <Activity mode={show
        ? "visible" : "hidden"}>
        <Sidebar />
      </Activity>

      <button onClick={() =>
        setShow(!show)
      }>
        Toggle
      </button>
    </>
  );
}
✓ Form inputs preserved
✓ Scroll position kept
✓ No re-mount cost
✓ Effects properly cleaned up

Actions

The Old Pattern (React 18)

  • onSubmit handler on the form
  • event.preventDefault() to stop page reload
  • fetch() to send data
  • useState for loading, error, and success state
  • Try/catch for error handling
Lots of boilerplate for every single form.
Data

React 18: A Simple Form

function UpdateName() {
  const [name, setName] = useState('');
  const [error, setError] = useState(null);
  const [isPending, setIsPending] = useState(false);

  const handleSubmit = async (e) => {
    e.preventDefault();
    setIsPending(true);
    const error = await updateName(name);
    setIsPending(false);
    if (error) setError(error);
  };

  return (
    <form onSubmit={handleSubmit}>
      <input value={name} onChange={(e) => setName(e.target.value)} />
      <button disabled={isPending}>Update</button>
      {error && <p>{error}</p>}
    </form>
  );
}
3 useState calls + manual async handling for one form

Actions

The New Pattern (React 19)

  • Pass an async function to the action prop on <form>
  • React handles the async transition automatically
  • No need for event.preventDefault()
  • No manual loading state management
  • Receives FormData directly

Key insight

Actions work with useTransition under the hood — the UI stays responsive during submission.

Data

React 19: Form Actions

<form action={async (formData) => {
  const error = await updateName(
    formData.get("name")
  );
  if (error) {
    // handle error
  }
}}>
  <input type="text" name="name" />
  <button type="submit">Update</button>
</form>
✓ No preventDefault
✓ No useState
✓ No try/catch
✓ Native FormData

useActionState

Manages the full lifecycle of a form action: previous state, current result, pending state

const [state, formAction, isPending] = useActionState(fn, initialState)

What it replaces

  • Manual useState for form results
  • useReducer patterns for form state
  • Custom pending state tracking

Returns 3 values

  • state — current action result
  • formAction — pass to <form action>
  • isPending — submission in progress
Login forms — server validation errors + pending state in one hook
Multi-step wizards — previous state tracks completed steps
CRUD forms — replaces useReducer + useState + useTransition
Data

useActionState in Practice

function ChangeName() {
  const [error, submitAction, isPending] = useActionState(
    async (previousState, formData) => {
      const error = await updateName(formData.get("name"));
      if (error) return error;
      redirect("/path");
      return null;
    },
    null,
  );

  return (
    <form action={submitAction}>
      <input type="text" name="name" />
      <button disabled={isPending}>Update</button>
      {error && <p>{error}</p>}
    </form>
  );
}
Data

useFormStatus

Read the parent form's submission state from any child component — no prop drilling

const { pending } = useFormStatus() 
function SubmitButton() {
  const { pending } = useFormStatus();
  return (
    <button type="submit" disabled={pending}>
      {pending ? "Saving..." : "Save"}
    </button>
  );
}
Why — Without it, you must pass isPending as a prop from every parent form to every submit button
When — Design system buttons, shared form components, any reusable UI inside a <form>
Data

useOptimistic

Update the UI instantly while waiting for the server response

const [optimistic, addOptimistic] = useOptimistic(state, updateFn)

How it works

  • Shows the expected result immediately
  • Reverts automatically if the action fails
  • Merges with real data when server responds

Great for

  • "Like" buttons
  • Comment submission
  • Todo lists
  • Any action where instant feedback matters
Data

useOptimistic in Practice

const [optimisticMessages, addOptimisticMessage] = useOptimistic(
  messages,
  (state, newMessage) => [
    ...state,
    { text: newMessage, sending: true }
  ]
);

// In the form action:
async function sendMessage(formData) {
  const message = formData.get("message");
  addOptimisticMessage(message);  // UI updates instantly
  await deliverMessage(message); // Server call
}
Demo tip: Add a setTimeout(resolve, 2000) to exaggerate the delay — shows the instant update vs real data arriving.

The use Hook

use(promise) — Reading Promises

Read async data directly in render. Integrates with Suspense for loading and ErrorBoundary for errors.

import { use, Suspense } from 'react';

function Message({ messagePromise }) {
  const content = use(messagePromise);
  return <p>Here is the message: {content}</p>;
}

// Usage with Suspense
<Suspense fallback={<p>Loading...</p>}>
  <Message messagePromise={fetchMessage()} />
</Suspense>
Replaces — useEffect + useState + loading/error booleans for data fetching
Reach for it when — data passed as a promise from parent, conditional async reads, Suspense-first architecture
Avoid — don't create promises inside render — pass them as props or lift to a parent
Data

use(Context) — Conditional Context

The key differentiator: use is NOT bound by the Rules of Hooks

useContext

  • Must be at the top level
  • Cannot be inside if / loops
  • Cannot be after early returns

use

  • Can be called in conditionals
  • Can be called in loops
  • Can be called after early returns
function HorizontalRule({ show }) {
  if (show) {
    const theme = use(ThemeContext);  // This is valid!
    return <hr style={{ color: theme.color }} />;
  }
  return false;
}

Server Components — In Practice

No useState, no useEffect, no client bundle cost — just async rendering

// This component never ships JS to the browser
async function ProductPage({ id }) {
  const product = await db.products.findUnique({ where: { id } });
  const reviews = await db.reviews.findMany({ where: { productId: id } });

  return (
    <main>
      <h1>{product.name}</h1>
      <p>{product.description}</p>
      <AddToCartButton id={id} />  {/* Client Component */}
      <ReviewList reviews={reviews} />
    </main>
  );
}
Use for — data-heavy pages, product listings, dashboards, blog content
Skip for — interactive widgets, forms, anything needing useState / useEffect

Server Actions

  • Async functions marked with "use server" that run on the server
  • Callable from client components — the framework creates a network endpoint behind the scenes
  • Tightly integrates with <form action={...}> for data mutations
  • Replaces custom REST/GraphQL endpoints for many common cases

Think of it as

RPC for React — call server functions as if they were local

Framework support

Next.js App Router, Remix, and other React frameworks

Data

Server Actions in Practice

// This is a Server Component
import Button from './Button';

function EmptyNote() {
  async function createNoteAction() {
    'use server';

    await db.notes.create();
  }

  return <Button onClick={createNoteAction} />;
}
✓ No API route file
✓ No fetch call
✓ Type-safe end to end
Data

Server Actions — Important Points

Security Warning

Server Actions are exposed as public HTTP endpoints.
Always validate inputs and authenticate requests — treat them like any API route.

Not a Replacement for Everything

Complex APIs still need dedicated endpoints.
Server Actions are best for mutations and simple data fetches.

Framework Required

Server Components and Actions need framework infrastructure.
Next.js 14+, Remix, or other compatible frameworks.

Migration Strategy

🚀 How to Start

  • Upgrade: npm install react@19 react-dom@19
  • Run codemods for forwardRef and Context.Provider
  • Enable Compiler on one non-critical page first
  • Check framework support for Server Components

📋 How to Proceed

  • Adopt features one at a time, not all at once
  • Low-risk wins first: metadata, ref cleanup, Context
  • Then Actions / useActionState for forms
  • Server Components for data-heavy pages

⚠️ Risks & Gotchas

  • Compiler requires Rules of React — run health checks
  • Server Components need framework + bundler support
  • <Activity> is experimental — API may change
  • Some third-party libraries may lag behind

  • Codemods — automated transforms for forwardRef, Context.Provider, and more
  • Compiler health checks — validate Compiler compatibility
